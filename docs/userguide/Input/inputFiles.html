Input guide#

This section introduces the NIAMH-MPCD input files.

A guide to all input parameters is provided on Github here. Furthermore, the Usage section in this user guide give detailed explanations on how to interpret each parameter and the Tutorials give examples of how to set up input files for specific simulations.

All input parameters must be listed in a single JSON file. Every parameter in the simulation can be fed in with a corresponding name/value pair. The name, or “tag”, uniquely represents each parameter, is case sensitive, but can be given in any order within the json file.

There are some exceptions to this rule, namely overrides. Overrides do not correspond to simulation input parameters, but instead will override some other behaviour in a helpful manner.

If a tag is not specified, then that parameter will assume the default value. This means that not every parameter needs to be explicitly set in the input file.

Input json files#

The following is a simple example of an input.json file that sets up a trivial simulation with periodic boundary conditions and a single species of thermalized fluid particles. This example illustrates the basic input structure.

{
    "comment":          "SIMULATION-SCALE INPUTS",
    "domain":           [30, 30],
    "simSteps":         2000,
    "comment":          "FLUID PARTICLE SPECIES INPUTS",
    "species":
    [
        {
            "dens":         20
        }
    ],
    "comment":          "BOUNDARY CONDITION INPUTS",
    "domainWalls":      1,
    "comment":          "OUTPUT",
    "energyOut":        10
}

Note

All arguments are case sensitive.

Understanding the Script#

Here is the breakdown of the input script.

{
    "comment":          "SIMULATION-SCALE INPUTS",

All arguments are given within " ":. The order does not matter (with some exceptions for overrides) The input arguments are contained between curly brackets {} — The first line is the opening bracket.

Comments can be label aspects of input files. We have added support for comments in the .json by using comment tags. If your tag is one of “c”, “comment”, “//”, or “#” then it will be ignored by the parser. Comment tags can be repeated (despite it being invalid .json) and not cause issues with the simulation.

"domain":           [30, 30],
"simSteps":         2000,

The system wide inputs set here are "domain" and "simSteps". The "domain" array sets the system size and "simSteps" sets the number of simulation steps (or iterations) that are performed.

Note

If the "domain" has a length of 2, the system is assumed to be 2D. If it has a length of 3, it is 3D. Other lengths are not permitted. A 2D system actually corresponds to a [30,30,1] domain (as opposed to a [30,30,0] domain).

Warning

Setting the domain does not set the boundary conditions. That must be done separately.

Note

These values correspond to their default values but users are expected to run much larger and longer simulations when doing research.

"species":
[
    {
        "dens":         20
    }
],

Next are the fluid particle properties. The "dens" override argument is used to set the average number of MPCD particles per accessible MPCD cell. Here, the number density is set to the default value of 20.

"domainWalls":      1,

Next the boundary conditions are set. In this example, simple periodic boundary conditions are required and therefore an override can be employed. "domainWalls": 1 creates four periodic boundaries around the domain in 2D and six in 3D.

Note

Additional boundaries can be added after the "domainWalls" override. In this way, solid obstacles or mobile colloids can be suspended within the domain with PBCs or impermeable/no-slip BCs.

    "energyOut":        10
}

Finally the output is specified. The default output is 0 (meaning that the quantity is not outputted) for all outputs, except "synopsisOut", which should always be outputted. Specifying "energyOut": 10 tells NAIMH-MPCD to write the global energy data every 10 timesteps.

Note

Times in the input file (such as run duration "simSteps" and outputs like "energyOut") are given in terms of the number of iterations \(\delta t\), not time in simulation units \([\tau]\) (see Derived units).

The input arguments are contained between curly brackets {} — The final line is the closing bracket.

Input Tables#

The table below lists all tags, their types, their default values, their description and links to where they are explained in this userguide. Each tag is linked to a description of its physical meaning in the Usage section. Tags will generally be named the same as they are named in the code (in the various input structs), unless there is a good reason to change it for legibility or clarity. When in doubt about which tags correspond to which variable names in the code, please consult SRDclss.h, definitions.h and the subroutine readJson() in read.c.

Some tags will take an array of custom objects (for example, species and boundary conditions). These are each defined in their own table following the main one.

All tags must be in a single JSON file.

Main overrides#

Tag

Type

Override param

Description

domainWalls

int

BC

This override will add extra BCs to the simulation, on top of the declared ones, on the domain walls. If set to 1, it places PBCs, and if set to 0 it places solid walls.

checkpointTimerOut

float

checkpointOut

This override enables checkpointing, but puts it on a timer. It will checkpoint every X hours, where X is specified by this parameter.

Tag

Type

Override param

Description

double

pop

This override will set a given species population to correspond to a cell density. This sets pop to the volume of the whole system (i.e. volume of the domain, excluding any excluded regions due to BCs) times this given value.

BC tag table#

These tags must go inside the "BC" array tag. Multiple BCs are allowed in the array, enclosed by square brackets ([]). The maximum number of BCs MAXBC is hardcoded and read by the compiler at compile time from mpcd/headers/definitions.h.

Curly brackets ({}) should enclose each BC and they should be separated by a comma (,).

Unlike the previous tags, if you declare a new BC object then there are some tags that must be declared. These are noted as NECESSARY in the default value column.
